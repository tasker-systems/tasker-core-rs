# TAS-111: Standardize and Optimize cargo-make Configuration

## Summary
Standardize cargo-make task definitions across the tasker-core workspace using the `extends` pattern to improve consistency, reduce duplication, and enable better task delegation from root to crate-specific Makefile.toml files.

## Problem Statement
Current cargo-make configuration has several issues:
1. **Inconsistent task naming** across workspace (e.g., `check-rust` vs `rust-check` in different contexts)
2. **Duplication** of similar patterns across worker makefiles
3. **No use of `extends`** missing opportunity for inheritance and delegation
4. **Ambiguous action names** - "check" performs different operations in different contexts
5. **No shared base tasks** for common patterns
6. **Inconsistent clean implementations** across workers

## Solution Design

### Action Nomenclature Standardization

Define clear, unambiguous actions that will be consistent across all crates:

| Action | Description | Operations |
|--------|-------------|------------|
| `build` | Compile/build artifacts | Standard build |
| `build-release` | Production build | Optimized/release build |
| `clean` | Remove build artifacts | Clean target dirs, caches |
| `format` | Check code formatting | Verify formatting standards |
| `format-fix` | Auto-fix formatting | Apply formatting changes |
| `lint` | Run linters | Static analysis, clippy, etc. |
| `lint-fix` | Auto-fix linting issues | Apply safe lint fixes |
| `test` | Run unit tests | Crate-local unit tests |
| `test-integration` | Integration tests | Tests requiring services |
| `test-e2e` | End-to-end tests | Full system tests (root only) |
| `test-docs` | Documentation tests | Rust doctests |
| `run` | Run service | Source .env, build, run foreground |
| `setup` | Install dependencies | Language-specific deps |
| `setup-system` | System setup | DB creation, migrations, etc. |

### Shell Scripts

#### `cargo-make/scripts/setup-nextest.sh`
```bash
#!/usr/bin/env bash
set -euo pipefail

if [ ! -f ".config/nextest.toml" ]; then
    echo "üì¶ Setting up nextest profiles..."
    mkdir -p .config
    cat > .config/nextest.toml << 'EOF'
[profile.default]
# Default profile for local development
failure-output = "immediate"
test-threads = "auto"
retries = { backoff = "exponential", count = 2 }

[profile.ci]
# CI profile with JUnit output (aligns with CI workflows)
failure-output = "final"
failure-summary = "json"
test-threads = 4
junit = { path = "target/nextest/ci/junit.xml" }
EOF
    echo "‚úÖ Nextest profiles configured"
else
    echo "‚úÖ Nextest profiles already exist"
fi
```

## Migration Strategy

### Backward Compatibility

No backward compatibility is provided for old task names. This is a greenfield, pre-alpha open source project and we are building out tooling as a net-new feature.

### CI Workflow Migration
1. First, ensure all cargo-make tasks work locally
2. Update CI workflows one job at a time
3. Keep existing scripts in `.github/scripts/` working
4. Gradually migrate CI to use `cargo make ci-*` tasks

### Rollback Plan
If issues arise:
1. Root `Makefile.toml` changes can be reverted independently
2. Worker Makefiles maintain their current functionality
3. CI workflows can fall back to direct commands
4. Git revert of the cargo-make/ directory removes all new infrastructure

## Risks & Mitigations
- **Risk**: Breaking existing workflows
  - **Mitigation**: Implement incrementally, maintain aliases for old task names
- **Risk**: Parallel execution conflicts
  - **Mitigation**: Only parallelize read-only operations (format, lint)
- **Risk**: Complex inheritance chains
  - **Mitigation**: Keep to 2-level depth max (base ‚Üí language-specific ‚Üí crate-specific)
- **Risk**: Path resolution issues in extended files
  - **Mitigation**: Use `${WORKSPACE_ROOT}` (defined in workspace-config.toml) instead of `${CARGO_MAKE_WORKING_DIRECTORY}`
